# 双端协作提示词模板

> 先填任务卡，再复制这里的提示词。
> 把占位符替换成实际项目内容后使用。

## 1. 给规划 / 复核方：冻结边界

```text
先不要实现代码。
请先帮我整理这次任务的协作边界，只输出以下内容：

1. 目标
2. 非目标
3. 允许改动范围
4. 禁止改动范围
5. 需要冻结的契约（接口 / 数据结构 / 配置项 / 错误语义 / 依赖边界）
6. 主写与复核角色建议
7. 最低验收标准
8. 风险与未验证项

任务背景：
[填写]
```

## 2. 给主写方：功能实现

```text
请按已经冻结的边界实现，不要只给方案。

任务目标：
[填写]

非目标：
[填写]

允许修改：
[填写]

禁止修改：
[填写]

冻结契约：
- 接口：[填写]
- 数据结构：[填写]
- 配置项：[填写]
- 错误语义：[填写]
- 依赖边界：[填写]

要求：
1. 优先复用现有实现模式，不凭空重写结构。
2. 只修改允许范围内文件。
3. 如果必须改公共文件，先说明原因。
4. 完成后做最小必要验证。
5. 输出改动文件、验证结果、未验证项、已知风险。

验收标准：
[填写]
```

## 3. 给主写方：缺陷修复

```text
请直接修复这个问题，并尽量收敛到最小影响面。

问题现象：
[填写]

预期行为：
[填写]

怀疑范围：
[填写]

允许修改：
[填写]

限制：
1. 不要顺手重构无关模块。
2. 优先修复根因，不要只做表面兜底。
3. 如果涉及契约变化，明确说明影响面。

完成后请输出：
1. 根因
2. 改动点
3. 验证方式
4. 未验证项
5. 风险
```

## 4. 给主写方：重构

```text
请在不改变外部行为的前提下，对指定模块做重构。

重构目标：
[填写]

必须保持不变：
- [填写]

允许优化：
- [填写]

禁止事项：
- 改接口契约
- 改无关模块
- 删除现有测试而不补验证

请先确认影响面，再实施，并在结果里写明：
1. 行为保持不变的依据
2. 修改文件
3. 验证结果
4. 未验证项
5. 剩余风险
```

## 5. 给复核方：工程 review

```text
请按工程 review 的方式审查下面这项改动，重点找风险，不要只做总结。

任务目标：
[填写]

变更概要：
[填写]

关键文件：
- [填写]

请重点审查：
1. 契约是否漂移
2. 是否越过允许改动范围
3. 是否存在边界条件遗漏
4. 是否可能引入回归
5. 测试是否足够
6. 文档是否需要同步

输出格式：
1. Findings
2. Open Questions
3. 建议补充的验证项
```

## 6. 给复核方：生成验收清单

```text
请基于以下任务生成一份详细验收清单，按以下五类输出：
1. 正常场景
2. 异常场景
3. 兼容性
4. 配置
5. 回滚

任务目标：
[填写]

技术栈：
[填写]

接口或输入输出：
[填写]

变更范围：
[填写]
```

## 7. 双端启动会

```text
这是一次协作开发任务，请先不要直接扩散实现，先统一边界。

任务目标：
[填写]

技术栈：
[填写]

允许修改：
[填写]

禁止修改：
[填写]

请先产出：
1. 任务拆分
2. 文件归属建议
3. 需要冻结的接口 / 数据 / 配置项
4. 验收标准
5. 风险点

然后再给出：
- 主写方建议
- 复核方建议
- 各自应使用的提示词
```

## 使用建议

- 需求不清晰时，先用“冻结边界”模板
- 需求明确时，主写方直接用“功能实现 / 缺陷修复 / 重构”模板
- 提交前用“工程 review”或“生成验收清单”模板
- 不要给两个执行方同一段模糊提示词
