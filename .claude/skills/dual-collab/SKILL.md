# Dual Collaboration

## Purpose

这个 skill 用于把模糊开发需求整理成可执行的协作任务。

它的目标不是直接实现代码，而是先建立一套稳定协作流程，减少以下问题：

- 接口契约漂移
- 文件归属冲突
- 范围不断扩散
- 验收口径不一致
- “代理说完成了”但工程没有闭环

适用对象包括：

- Claude Code 与 Codex 协作
- 多个 Claude / agent 协作
- 一人主写、一人 review
- 单人开发时先规划、后实现、再复核

---

## When to Use

在以下场景优先使用本 skill：

- 新功能开发前
- 复杂 bug 修复前
- 重构前
- 多文件修改前
- 涉及接口、配置、数据结构变化前
- 多代理 / 多人协作前
- 提交前需要统一验收口径时

如果只是极小修改，例如单行修复、明显 typo、小范围改名，可不必完整套用。

---

## Core Principle

协作开发默认遵循以下原则：

1. 一个任务只保留一个主写者
2. 一个文件同一时间只保留一个主修改人
3. 先冻结边界，再开始实现
4. 先冻结契约，再并行推进
5. 先定义验收标准，再开始编码
6. 先写清风险和未验证项，再宣称完成
7. 不把“已经实现”当成“已经验收”

---

## What “Freeze Boundary” Means

冻结边界，指的是在实现前先明确：

- 这次要做什么
- 这次不做什么
- 允许改哪些文件 / 目录
- 禁止改哪些文件 / 目录
- 这次如何算完成

最小边界卡应至少包含：

```text
目标：
非目标：
允许改动：
禁止改动：
验收标准：
```

---

## What “Freeze Contract” Means

冻结契约，指的是在实现前先明确这次是否允许变更：

- 接口路径、入参、出参
- 数据结构
- 配置项名称、默认值、覆盖方式
- 错误语义 / 错误码
- 依赖边界

如果允许变化，必须提前写清影响面。
如果没有明确允许，默认不改。

---

## Recommended Role Split

默认角色建议如下：

### 主写方
更适合负责：

- 落代码
- 搜索实现位置
- 修改文件
- 跑验证命令
- 汇总改动结果

### 复核方
更适合负责：

- 目标 / 非目标澄清
- 边界冻结
- 契约检查
- 风险补充
- 测试缺口检查
- 交付说明与验收清单

---

## Recommended Workflow

使用本 skill 时，按下面顺序工作：

### Step 1: Clarify Goal
先明确：

- 目标
- 非目标
- 当前问题
- 预期结果

### Step 2: Freeze Scope
明确：

- 允许修改范围
- 禁止修改范围
- 是否允许改公共模块
- 是否允许改接口 / 配置 / 数据结构 / 依赖

### Step 3: Assign Ownership
明确：

- 谁主写
- 谁复核
- 哪些文件归谁主改
- 是否需要 branch / worktree 隔离

### Step 4: Generate Prompts
根据任务类型生成：

- 给主写方的实现提示词
- 给复核方的 review / 风险检查提示词

### Step 5: Define Acceptance
明确：

- 验证命令
- 正常路径
- 异常路径
- 已验证 / 未验证
- 风险
- 回滚方式（如适用）

## Common Collaboration Modes

### Mode A: Design First, Implement Second
适合：

- 需求还不稳定
- 存在多种实现路径
- 契约可能变化
- 风险较高

方式：

- 先由复核/规划方冻结边界和契约
- 再由主写方实现
- 最后再复核和补验收项

### Mode B: Implement First, Review Second
适合：

- 需求非常明确
- 改动范围可控
- 需要快速推进

方式：

- 主写方先实现
- 复核方做 review、补风险与测试检查

### Mode C: Parallel by Ownership
适合：

- 文件边界很清楚
- 主链路与测试/文档可拆开推进

方式：

- 一方主改业务
- 一方主改测试 / 文档 / 验收材料

---

## Anti-Patterns

使用本 skill 时，避免以下做法：

- 给两个执行方同一个模糊需求
- 让两个执行方同时自由修改同一批核心文件
- 跳过任务卡直接开写
- 在未同步的情况下修改契约
- 把功能实现、顺手重构、全量格式化、补文档混成一次改动
- 把“我觉得差不多了”当成完成
- 把“代理说完成了”当成验收完成

## Prompt Generation Rules

### 给主写方的提示词应包含
- 明确目标
- 非目标
- 允许修改范围
- 禁止修改范围
- 冻结契约
- 验收标准
- 输出要求

### 给复核方的提示词应包含
- 任务目标
- 变更概要
- 关键文件
- 重点检查项
- 输出格式（Findings / Open Questions / Suggested Checks）

---

## Validation Rules

默认要求至少覆盖：

- 一个正常场景
- 一个异常场景
- 核心配置可读取
- 文档与实际行为一致

交付时必须显式区分：

- 已验证
- 未验证
- 已知风险

如果任务涉及部署、迁移、配置、脚本写操作，还要补：

- 影响面
- 回滚方式
- 是否会覆盖已有数据 / 环境

---

## Handoff Rules

一个执行方把结果交给另一个执行方时，至少应同步：

```text
目标:
当前状态:
已改文件:
受影响文件:
契约变化:
配置变化:
数据变化:
已验证:
未验证:
风险:
下一步建议:
```

---

## How to Respond

当用户请求“帮我按这套协作方式整理任务”时：

- 不要直接开始实现
- 先输出任务卡和角色分配
- 如果需求不清晰，优先补齐边界和契约
- 如果需求清晰，再输出针对主写方和复核方的提示词
- 最后补上验收清单和风险

当用户请求“帮我 review 现有改动”时：

- 不要重写实现
- 优先检查契约漂移、边界越界、异常路径缺失、验证不足、文档遗漏

### Local backend skill collaboration

当前仓库如涉及后端实现或评审，建议在本 skill 冻结边界后，再按任务类型调用本地技能：

- `kb-spring-boot-rag-patterns`：Spring Boot + RAG 主后端实现/评审
- `mybatis-plus-patterns`：MyBatis-Plus 数据访问与查询边界
- `kb-api-contract-review`：接口契约、一致响应与错误语义审查
- `kb-test-quality`：Controller / Service / RAG / Redis / Milvus 测试设计与评审
- `kb-logging-patterns`：traceId、索引、缓存、Milvus、降级日志设计
- `kb-java-code-review`：本项目 Java 改动的结构化 review

换言之：
- `dual-collab` 负责冻结边界、职责划分、交接格式、验收清单
- 本地后端 skills 负责在既定边界内给出项目适配的实现与评审规则

---

## Goal of This Skill

这个 skill 的最终目标不是生成更多文档，而是让每次开发都更像一个有边界、有分工、有验证的工程任务，而不是一次临时聊天。
