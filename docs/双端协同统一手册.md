## 1. 怎么使用这份文档

这份文档不是让你把同一段模糊需求同时发给 `Claude Code` 和 `Codex`。

正确用法是：

1. 先用任务模板把本轮任务实例化
2. 再指定主写代理和复核代理
3. 再从提示词模板中复制适合的提示词发给两个端
4. 最后用统一验证结果和 PR 检查清单收口

也就是：

**手册定规则，任务模板定本轮边界，提示词模板定职责分工，PR 模板定交付与验收。**

## 2. 一页速查版

### 2.1 先记住一句话

不是“同时打开 `Claude Code` 和 `Codex`，把同一份手册都发给它们”就可以。

正确方式是：

1. 先把本轮任务写成一张任务卡
2. 再分配角色：谁主写，谁复核
3. 再给两个代理发送不同职责的提示词
4. 最后用统一验证结果和 PR 清单收口

### 2.2 日常最小流程

#### Step 1：先写任务卡
至少写清这 5 项：

- 目标
- 非目标
- 允许改动范围
- 禁止改动范围
- 验收标准

最小模板：

```md
目标：
非目标：
允许改动：
禁止改动：
验收标准：
```

#### Step 2：再分角色
只保留两个角色：

- **主写代理**：负责落代码
- **复核代理**：负责边界、风险、测试缺口、契约漂移检查

默认建议：

- **需求清楚**：`Codex` 主写，`Claude Code` 复核
- **需求不清楚**：`Claude Code` 先拆解和冻结边界，`Codex` 再实现

#### Step 3：分别发不同提示词

- 发给 `Claude Code`：边界冻结、风险分析、review、验收清单
- 发给 `Codex`：实现、改文件、跑验证、汇总结果

#### Step 4：最后统一收口
不要只看“代理说已经完成”。

必须至少看：

- 改动文件列表
- 已执行的验证命令
- 已验证项
- 未验证项
- 已知风险

### 2.3 两种最常用场景

#### 场景 A：需求还不清楚
流程：
1. 先写任务卡
2. 发给 `Claude Code`：冻结目标、非目标、文件归属、契约、风险、验收口径
3. 发给 `Codex`：按冻结后的边界实施
4. 再发给 `Claude Code`：做 review 和补充验收项

适用情况：
- 新功能
- 多种实现路径都可行
- 接口 / 配置 / 数据结构可能受影响
- 范围还不稳定

#### 场景 B：需求已经很清楚
流程：
1. 先写任务卡
2. 指定 `Codex` 为主写代理
3. 指定 `Claude Code` 为复核代理
4. `Codex` 实现并输出验证结果
5. `Claude Code` 做 review 和验收补缺

适用情况：
- 需求很明确
- 已知修改范围
- 改动主要是代码落地
- 需要快速推进

### 2.4 最短可复制提示词

#### 先发给 Claude Code：冻结边界

```text
先不要实现代码。
请先帮我冻结本轮任务的边界，只输出以下内容：
1. 目标与非目标
2. 允许改动范围
3. 禁止改动范围
4. 接口/配置/数据结构是否需要冻结
5. 风险点
6. 最低验收标准
```

#### 再发给 Codex：实施

```text
请按已经冻结的边界实现。
要求：
1. 只修改允许范围内的文件
2. 不要扩展需求
3. 不要顺手重构无关模块
4. 如果必须改公共文件，先说明原因

完成后输出：
1. 改动文件列表
2. 已执行验证命令
3. 已验证项
4. 未验证项
5. 已知风险
```

#### 最后发给 Claude Code：复核

```text
请基于实现结果做工程 review，不要重新实现。
重点检查：
1. 是否有契约漂移
2. 是否越过允许改动范围
3. 是否遗漏异常路径
4. 是否存在测试缺口
5. 是否满足验收标准
```

### 2.5 日常不要这样用

- 不要给两个代理发同一个模糊需求
- 不要让两个代理同时自由修改同一批核心文件
- 不要跳过任务卡直接开写
- 不要一边改契约一边不通知另一端
- 不要把“代理说完成了”当成完成
- 不要把功能实现、顺手重构、格式化、补文档混成一波提交

### 2.6 一句话口诀

**先任务卡，再分角色，再分提示词，最后统一验收。**

## 3. 核心原则

- [ ] 一个任务只保留一个“最终裁决口”
- [ ] 一个文件同一时间只保留一个“主修改人”
- [ ] 先冻结契约，再拆分实现
- [ ] 先定义完成标准，再并行开工
- [ ] 先列风险，再写代码
- [ ] 先自检，再交叉 review
- [ ] 不把“AI 已经看过”当成“工程已经闭环”

协同时最容易失控的不是代码量，而是以下三件事：
- [ ] 接口契约漂移
- [ ] 文件归属不清
- [ ] 测试和验收口径不一致

## 4. 两个端的默认定位

`Codex` 更适合承担：
- [ ] 直接基于仓库落代码
- [ ] 多文件精确修改
- [ ] 搜索调用链和影响面
- [ ] 执行编译、测试、启动、日志检查
- [ ] 收敛最终补丁和变更结果

`Claude Code` 更适合承担：
- [ ] 拆需求和梳理边界
- [ ] 设计接口契约和模块职责
- [ ] 枚举异常路径和回滚策略
- [ ] 做二次审查和反例检查
- [ ] 产出可复用的说明文档、任务单、评审清单

不要把这理解成硬性分工，而应理解为默认强项：
- [ ] 当任务是“方案重于落地”时，让 `Claude Code` 主导
- [ ] 当任务是“落地重于讨论”时，让 `Codex` 主导
- [ ] 当任务既大又杂时，拆成文件级或阶段级所有权

## 5. 协作模式选择

### 5.1 模式 A：Claude Code 主设计，Codex 主落地

适合：
- [ ] 新功能范围还不稳
- [ ] 有多种实现路径需要比较
- [ ] 兼容性和异常路径复杂
- [ ] 需要先梳理接口、数据结构、配置边界

流程：
- [ ] `Claude Code` 先给方案、边界、风险、任务拆分
- [ ] `Codex` 基于仓库逐步实现
- [ ] `Claude Code` 最后做复核和补验收点

### 5.2 模式 B：Codex 主实现，Claude Code 主评审

适合：
- [ ] 需求很明确
- [ ] 改动以代码落地为主
- [ ] 需要快速试错和本地验证
- [ ] 仓库已有比较稳定的结构和风格

流程：
- [ ] `Codex` 先改代码、跑命令、形成第一版结果
- [ ] `Claude Code` 审查风险、遗漏场景和交付说明
- [ ] `Codex` 根据 review 收敛最终修改

### 5.3 模式 C：双端并行，一端业务一端测试

适合：
- [ ] 改动面较大但文件边界清晰
- [ ] 业务实现和测试验收可以分开推进
- [ ] 需要压缩总交付时间

流程：
- [ ] 一个端负责业务代码
- [ ] 一个端负责测试、文档、联调脚本、验收项
- [ ] 在接口和文件归属冻结后并行推进

### 5.4 模式 D：双端并行，一端主线一端支线

适合：
- [ ] 主功能开发之外还有配置、迁移、监控、运维脚本等支线工作
- [ ] 不希望支线工作阻塞主线开发

流程：
- [ ] 一个端负责主链路代码
- [ ] 一个端负责文档、配置、部署脚本、监控、回滚说明

## 6. 开工前统一清单

- [ ] 明确任务标题
- [ ] 明确目标和非目标
- [ ] 明确目标分支
- [ ] 明确是否允许改动公共模块
- [ ] 明确是否允许改接口契约
- [ ] 明确是否允许改数据库结构
- [ ] 明确是否允许改配置项
- [ ] 明确是否允许改依赖版本
- [ ] 明确最低验证要求
- [ ] 明确谁负责最终合并
- [ ] 明确本轮主写代理是谁，谁负责 review / 补测试 / 补文档 / 风险检查
- [ ] 明确基线分支或 commit SHA，避免两端站在不同基线继续扩散

开工前至少冻结以下内容：
- [ ] 文件归属
- [ ] 接口契约
- [ ] 数据结构
- [ ] 配置项命名
- [ ] 验收口径
- [ ] 测试命令入口

### 冻结边界是什么意思

冻结边界就是在开工前先把“这次到底做什么、不做什么、谁能改哪里”定死。

最少写清：
- 本轮目标
- 本轮非目标
- 允许改哪些文件 / 目录
- 禁止改哪些文件 / 目录
- 验收标准

### 冻结契约是什么意思

冻结契约就是在开工前先说清楚这次是否允许变更以下内容：
- 接口路径、入参、出参
- 数据结构
- 配置项名称、默认值、覆盖方式
- 错误码或异常语义
- 依赖边界

如果允许变，就要提前写清影响面；如果不允许变，就默认不能顺手调整。

## 7. 文件归属、分支与 worktree 约定

强烈建议在开工前先分配“所有权”，不要让两个端同时改一片区域。

工作区隔离建议：
- [ ] 不同代理不要同时操作同一个工作目录
- [ ] 每个代理使用独立 branch，必要时再配独立 `git worktree`
- [ ] 分支命名可直接体现角色，例如 `feat/xxx-claude`、`review/xxx-codex`
- [ ] 禁止两个代理直接在同一分支连续改动同一批文件
- [ ] 如果必须在同一任务内并行推进，先冻结契约，再按模块或文件边界拆开

推荐做法：
- [ ] 公共基础文件只给一个端主改
- [ ] 业务主链路代码只给一个端主改
- [ ] 测试和文档可单独分配给另一个端
- [ ] 配置、部署、脚本也要单独定义归属
- [ ] 一旦需要改公共抽象，先暂停并重新分配

高风险公共文件示例：
- [ ] 基础配置文件
- [ ] 路由或入口文件
- [ ] 统一响应或错误码定义
- [ ] 公共组件库
- [ ] 公共类型定义
- [ ] 数据模型基类
- [ ] 数据库 schema / migration
- [ ] 构建脚本
- [ ] CI 配置
- [ ] Dockerfile
- [ ] 部署脚本

禁止事项：
- [ ] 禁止双端同时修改同一个核心文件
- [ ] 禁止一端修改契约，另一端继续按旧契约开发
- [ ] 禁止未同步就调整公共枚举、常量、状态机、环境变量名
- [ ] 禁止把未验证的临时结论当成最终约束

## 8. 标准协作流程

### 8.1 冻结边界
- [ ] 用一句话定义本轮目标
- [ ] 写清“不做什么”
- [ ] 写清本轮允许修改的目录或文件
- [ ] 写清本轮禁止修改的目录或文件

### 8.2 冻结契约
- [ ] 冻结接口
- [ ] 冻结数据模型
- [ ] 冻结错误语义
- [ ] 冻结配置项
- [ ] 冻结依赖边界

### 8.3 拆分任务
- [ ] 把工作拆成可以独立验证的小块
- [ ] 按文件或模块分配所有权
- [ ] 标记谁负责实现、谁负责复核、谁负责验收

### 8.4 并行执行
- [ ] 主实现端落地代码
- [ ] 评审端补测试、文档、review 清单
- [ ] 任何契约变化必须先同步，再继续编码
- [ ] 不要把功能实现、全量格式化、顺手重构、文档补充、测试补丁混在同一波改动里

### 8.5 自检
- [ ] 检查命名、依赖、导入、边界条件
- [ ] 检查异常路径
- [ ] 检查日志和可观测性
- [ ] 检查配置默认值和覆盖方式
- [ ] 检查安全风险
- [ ] 检查格式化是否限制在本次修改文件内，避免 diff 爆炸

### 8.6 交叉复核
- [ ] 由非主修改端做 review
- [ ] 优先看行为回归，不先纠结风格
- [ ] 优先看失败路径、兼容性、回滚成本
- [ ] 交接时同步基线分支或 commit SHA、改动文件列表、已执行命令、未解决问题、重点 review 点

### 8.7 联调验收
- [ ] 编译或构建通过
- [ ] 测试通过
- [ ] 启动或部署验证通过
- [ ] 主链路通过
- [ ] 关键异常链路符合预期
- [ ] 文档同步完成
- [ ] 验收以统一脚本命令和 CI 结果为准，不以代理口头说明为准

## 9. 常见任务类型协同建议

### 9.1 后端接口与服务

适用技术：
- [ ] Spring Boot
- [ ] Node.js / NestJS / Express
- [ ] Python / FastAPI / Django / Flask
- [ ] Go / Gin / Fiber

重点冻结：
- [ ] API 路径
- [ ] 入参和出参结构
- [ ] 错误码和异常语义
- [ ] 数据库字段和索引
- [ ] 事务边界
- [ ] 外部依赖超时和重试策略

推荐分工：
- [ ] `Claude Code` 先梳理契约、边界、状态流转
- [ ] `Codex` 实现 Controller、Service、Repository、测试
- [ ] `Claude Code` 检查异常路径、幂等性、兼容性

### 9.2 前端页面与交互

适用技术：
- [ ] React
- [ ] Vue
- [ ] Next.js
- [ ] Nuxt
- [ ] Svelte
- [ ] React Native
- [ ] Flutter Web 或混合项目中的前端部分

重点冻结：
- [ ] 页面状态定义
- [ ] 接口依赖和 mock 数据
- [ ] 组件边界
- [ ] 设计稿关键约束
- [ ] Loading、Empty、Error 三类状态
- [ ] 埋点和可访问性要求

推荐分工：
- [ ] `Claude Code` 先拆页面状态、组件树、交互清单
- [ ] `Codex` 实现页面、组件、数据请求、测试
- [ ] `Claude Code` 检查交互一致性、响应式、异常状态和文案

特别注意：
- [ ] 不要只检查 happy path
- [ ] 必须校验接口失败、空数据、慢加载、权限不足、重复提交
- [ ] 如果是设计还原任务，先冻结视觉约束再落代码

### 9.3 Python 脚本、自动化与数据任务

适用技术：
- [ ] Python CLI
- [ ] ETL
- [ ] Notebook 配套脚本
- [ ] 爬虫与数据清洗
- [ ] 批处理
- [ ] AI 推理或 embedding 管道

重点冻结：
- [ ] 输入输出格式
- [ ] 路径和目录约束
- [ ] 环境依赖
- [ ] 运行方式
- [ ] 失败重试和中断恢复策略
- [ ] 数据覆盖风险

推荐分工：
- [ ] `Claude Code` 先定义数据流、失败点、样例输入输出
- [ ] `Codex` 编写脚本、参数解析、日志、测试
- [ ] `Claude Code` 复核边界样本、空值、脏数据、幂等性

特别注意：
- [ ] 对文件系统和数据库的写操作要有保护措施
- [ ] 批处理任务必须先明确是否允许覆盖
- [ ] 大数据量任务必须关注性能和断点续跑

### 9.4 Node.js 工具链与中间层

适用技术：
- [ ] Node.js CLI
- [ ] 构建脚本
- [ ] BFF
- [ ] 网关
- [ ] Serverless 函数

重点冻结：
- [ ] 运行命令
- [ ] 环境变量
- [ ] 依赖版本
- [ ] 打包产物
- [ ] 与前后端的契约边界

推荐分工：
- [ ] `Claude Code` 梳理脚本职责、依赖顺序和失败处理
- [ ] `Codex` 改脚本、服务代码和校验逻辑
- [ ] `Claude Code` 复核可维护性、配置兼容性和回归风险

### 9.5 AI、RAG、检索与模型接入

### 12.2 填写说明

#### 1. 背景
- 写清问题现状，不要只写“优化一下”
- 写清目标行为，不要只写“修复 bug”
- 非目标必须写，避免双端越做越大

#### 2. 仓库与范围
- “基线分支 / commit”用于冻结双方开始工作的同一基线
- “主写代理”只保留一个，另一个代理默认负责 review、测试、文档或风险检查
- “分支 / worktree 方案”写清是否采用独立 branch、是否采用独立 `git worktree`
- “允许修改”写目录或文件
- “禁止修改”写不能动的公共区域
- 如果允许改数据库、配置、依赖版本，要明确写出来

#### 3. 技术栈与运行约束
- 写清语言、框架、构建工具、部署方式
- 写清本地验证依赖，比如数据库、缓存、消息队列、模型服务

#### 4. 冻结契约
- 接口契约包括路径、入参、出参、错误码
- 数据结构包括表结构、实体结构、前端状态结构、脚本输入输出格式
- 配置项包括环境变量、配置文件键名、默认值

#### 5. 文件归属
- 同一文件只能有一个主修改端
- 公共文件必须提前声明归属

#### 6. 职责分配
- `Codex` 更适合写实现、改文件、跑命令、做验证
- `Claude Code` 更适合做拆分、review、边界补充、文档沉淀

#### 7. 验收标准
- 验收必须可执行，不要写成抽象描述
- 例如“接口返回正确”不够，应写成“接口 A 正常返回 200，非法参数返回 400”

### 12.3 推荐场景

适合以下任务：
- 新增功能
- 修复复杂 bug
- 重构
- 多文件改动
- 加配置或外部依赖
- 部署变更
- 数据迁移

## 15. 常见失效模式

### 15.1 两个端都改了同一个文件
- [ ] 立即停下继续修改
- [ ] 先决定谁保留主版本
- [ ] 逐段核对业务语义和契约，而不是机械合并

### 15.2 契约先变了但没有同步
- [ ] 停止继续扩散实现
- [ ] 回到最近一次共同确认的契约版本
- [ ] 重新发一版冻结说明

### 15.3 只做了 happy path
- [ ] 回补异常路径、空数据、权限、超时、并发、幂等等场景
- [ ] 未补齐前，不要宣称已经完成

### 15.4 文档落后于代码
- [ ] 代码合并前必须同步文档
- [ ] 至少写清接口变化、配置变化、运行方式、风险说明

### 15.5 把公共修改伪装成局部修改
- [ ] 任何影响公共模块的改动都必须重新声明影响面
- [ ] 不要把“顺手改一下”带进最终提交
